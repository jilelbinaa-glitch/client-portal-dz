# إعداد الأمان — خطوة يدوية واحدة مطلوبة

بعد نشر `firestore.rules` الجديد و`bridge.js`، افتح **Firebase Console** ونفّذ 3 أشياء (مرة واحدة فقط):

## 1) فعّل طرق الدخول
`Authentication → Sign-in method` → فعّل:
- **Anonymous** (لزوار الموقع/تطبيق الجوال — تلقائي وشفاف تماماً)
- **Email/Password** (لموظفي المكتب)

## 2) أنشئ حساب دخول للمكتب
`Authentication → Users → Add user` → أدخل إيميل وكلمة سر لأي موظف تريد منحه صلاحية الدخول للوحة `architecture_pm.html`. انسخ الـ **UID** الذي يظهر له.

## 3) أضفه لقائمة الموظفين المخوّلين
`Firestore Database → Start collection` → اسم المجموعة `staff` → معرّف المستند = نفس الـ UID من الخطوة السابقة → أي حقل (مثلاً `name: "اسم الموظف"`) → Save.

كرّر الخطوتين 2 و3 لكل موظف إضافي.

---

## ماذا تغيّر تقنياً (ملخص)
- **قبل:** أي شخص يعرف رابط Firebase (project ID) يقدر يقرأ/يعدّل بيانات أي عميل، أو يسحب قائمة كل العملاء دفعة واحدة.
- **بعد:** معرفة "رقم التذكرة" (مثل رقم تتبّع شحنة) كافية لمتابعة طلب واحد بالذات — لكن لا أحد يقدر "يتصفّح" كل الطلبات إلا موظف مسجّل دخول فعلياً وموجود في `staff`.
- بنية البيانات تغيّرت قليلاً: `messages` و`notifications` الخاصة بعميل أصبحت مجموعات فرعية تحت طلبه (`requests/{ticket}/messages`)، ورقم التذكرة أصبح هو معرّف المستند نفسه بدل معرّف عشوائي.

## ⚠️ تنبيه مهم إن كان هناك بيانات فعلية منشورة حالياً
هذا التعديل **يغيّر بنية البيانات**. إذا كان المشروع يحتوي فعلاً على طلبات/رسائل حقيقية على Firebase الحالي (وليس فقط أثناء التطوير)، فهذه البيانات القديمة **لن تظهر** للتطبيق الجديد لأنها بالبنية القديمة (معرّفات عشوائية بدل رقم التذكرة، ومجموعات علوية بدل فرعية). إذا كان هذا ينطبق عليك، قلي وأجهّز لك سكريبت هجرة (Node.js + Firebase Admin SDK) قبل ما تنشر التعديلات على الإنتاج.
